# v0.1 Rebuild — Execution Plan (feat/v0.1-rebuild)

_Last updated: 2026-04-20_

## Purpose

Translate [22] Phase — Canonical Beta Foundation into an executable, per-commit
sequence on `feat/v0.1-rebuild`. This branch rebuilds the repo from its current
Vite + React SPA prototype state onto the locked canonical stack:
Next.js App Router + Supabase + Graphile Worker + OpenAI, owner-only beta.

This plan is scoped to Phase 22 only. Domain implementation (bookings,
availability blocks, requests, shifts, sharing, audit) does not land in this
branch — those are later phases on top of the foundation this branch establishes.

## Starting state (as of branch creation)

- Branch: `feat/v0.1-rebuild`, forked from `feat/mvp-shell` at commit `248c5e9`.
- Source branch commit `248c5e9` landed the shadcn teardown and reframed the
  README as a disposable SPA prototype.
- First commit on this branch: `3d346dd` — added wrangler 4.84.0 as devDependency.
  This is inconsistent with the locked stack (see "Immediate correction" below).
- `src/` currently contains the Vite SPA: 5 pages, mock data, per-page local
  state, hand-authored conflict flags. To be archived, not extended.
- `docs/` is source of truth and stays in place.

## Immediate correction

Cloudflare Workers is not part of the locked stack. The wrangler dependency
committed in `3d346dd` should be removed as the first move on this branch,
before any restructuring begins. If wrangler is retained for a specific narrow
use (e.g., a future share-link edge worker), `docs/source-of-truth.md` must be
updated to reflect the expanded locked stack before proceeding.

## Execution waves

The plan is organized into four waves. Each numbered slice is intended to be
one commit. Slice titles are in imperative form.

### Wave A — Repo restructure

The goal of Wave A is to end with Next.js App Router as the primary runtime
at the repo root, the Vite artifacts archived, and the canonical folder
structure laid out. No domain logic, no Supabase, no auth yet.

1. **Uninstall wrangler.** `npm uninstall wrangler` and commit.
2. **Archive the Vite SPA.** Move `src/`, `vite.config.ts`, `index.html`,
   `postcss.config.js`, `tailwind.config.js`, `tsconfig.app.json`,
   `tsconfig.node.json`, and `dist/` into `archive/vite-shell/`. Preserve
   Tailwind tokens and component shapes as reference input for the Next.js
   rebuild. Do not delete.
3. **Scaffold Next.js App Router at the repo root.** New `package.json` with
   `next`, `react`, `react-dom`, `typescript`, `tailwindcss`. Minimal
   `app/layout.tsx`, `app/globals.css`, `public/`. Verify `npm run dev`
   serves a hello page.
4. **Lay out the canonical folder structure.** Per Phase 22 §Repo
   restructuring plan:

   ```
   src/
     app/
       (public)/login/
       (app)/{agenda,coverage,intake,settings}/
       layout.tsx
       globals.css
     components/
     modules/{auth,bookings,availability-blocks,booking-requests,
              shift-occurrences,shift-templates,sharing,audit}/
     server/{db,repositories,services,policies}/
     lib/{supabase,config,openai,worker}/
   supabase/{migrations,seed,policies}/
   worker/{jobs,runner}/
   public/manifest.webmanifest + icons
   tests/{smoke,auth,routing}/
   ```

   Empty barrel exports only, no logic.

### Wave B — Config, auth, and protected shell

The goal of Wave B is that the owner can sign in via Supabase Auth on a
local dev server and land on a protected `/agenda` placeholder, with the
other three canonical routes (Coverage, Intake, Settings) reachable from
shell navigation.

5. **Config loader with env validation.** `src/lib/config/index.ts` validates
   required env vars at startup using zod or equivalent. `.env.local.example`
   lists every required key from Phase 22 §Environment and secrets plan, with
   no real values.
6. **Supabase clients.** `src/lib/supabase/{browser,server,admin}.ts`.
   Service-role key limited to server-only paths.
7. **Owner-only Supabase Auth.** Email magic-link login at `/login`, server-
   side session validation, middleware that redirects unauthenticated traffic
   from `(app)/*` to `/login`, `APP_ALLOWED_EMAILS` allowlist gate. No public
   signup. No self-serve invites.
8. **Protected shell.** `src/app/(app)/layout.tsx` with mobile-first
   navigation to Agenda / Coverage / Intake / Settings. Four route
   placeholders. Authenticated owner lands on `/agenda` by default. Nav must
   enforce Agenda = Bookings only and Coverage = Shift Occurrences only
   — those labels and separation cannot blur even in placeholder state.
9. **Role scaffolding.** `src/server/policies/` with `dj_owner` as the active
   role and `manager_lite` reserved but not surfaced in UI. All role decisions
   server-side. No client-trusted role logic.

### Wave C — Backend foundation

The goal of Wave C is that database, worker, OpenAI, and health checks are
all wired at the infrastructure level so later phases can add domain logic
without replacing the foundation.

10. **Supabase migrations skeleton.** `supabase/migrations/` with the first
    migration reserving `schedule_commitments` as a first-class table. Phase
    22 is explicit: it must be treated as a first-class normalized backend
    surface from the start, even if full business logic lands later. Reserve
    the table and any obvious columns (actor/subject/effective window,
    state-source enum); full schema can evolve in later phases.
11. **Graphile Worker runtime.** `worker/` with a working graphile-worker
    process connected to `GRAPHILE_WORKER_DATABASE_URL`, plus a `noop` job to
    prove wiring end-to-end. Deploy target for the worker must support a
    long-running process — see Risks.
12. **OpenAI adapter.** `src/lib/openai/client.ts`, server-only, reads from
    `OPENAI_API_KEY`. Adapter shape only; no extraction flow yet.
13. **Health endpoint.** `/api/health` returning booleans for app boot,
    database reachability, auth config presence, and worker config presence.

### Wave D — PWA, deploy, and acceptance

The goal of Wave D is that the private beta is deployable online, installable
as a PWA, and passes Phase 22's Definition of Done as the branch-merge gate.

14. **PWA baseline.** `public/manifest.webmanifest`, app icons, viewport/meta,
    theme color, cautious service worker strategy with versioned update path.
    No aggressive offline caching of mutable operational data.
15. **Environment separation and deploy.** Three environments —
    `local`, `preview`, `private-beta-production`. Separate Supabase projects.
    Preview must be isolated from live owner data or disabled entirely.
    Secrets via platform-managed stores; no `.env` files in version control.
    Next.js host plus a worker-capable host (Fly, Railway, or Render) for
    the Graphile Worker process.
16. **Phase 22 acceptance pass.** Run the Definition of Done checklist (14
    items) from Phase 22 §Definition of done. Close any gaps. This is the
    gate for merging `feat/v0.1-rebuild` back to `main` or to a downstream
    integration branch.

## Rough sizing

- Wave A: ~2–4 days (migration is heaviest part).
- Wave B: ~4–7 days (auth + shell is the most interactive piece).
- Wave C: ~3–5 days (mostly wiring, worker deploy adds uncertainty).
- Wave D: ~3–5 days (polish + deployment config).

Total: 2–4 weeks of focused solo work.

## Risks

- **Worker hosting topology.** Graphile Worker needs a persistent process;
  Vercel alone does not satisfy this. The Next.js host and worker host will
  likely be separate services. Flagged early in Phase 22 as the first
  operational blocker.
- **Preview/production data isolation.** Preview deployments must not touch
  the private-beta Supabase project. If isolation cannot be guaranteed,
  disable preview deploys rather than compromise.
- **`schedule_commitments` reservation.** Deferring this past Wave C would
  force a schema rewrite in later phases. Reserve it as a real table in the
  first migration, even minimally.
- **Design-language loss.** The Vite SPA has Tailwind tokens and component
  shapes worth porting. Archive it rather than delete it, and reference it
  during Wave B shell work.
- **Client role logic drift.** Any role check written into client components
  during placeholder work will need to be ripped out. Keep role decisions
  server-side from the first auth-aware commit.
- **Branch drift from `main` during rebuild.** Because this is a 2–4 week
  branch, `main` may accept unrelated hotfixes. Rebase regularly rather
  than merge, to keep the rebuild history linear.

## Out of scope for this branch

- Any domain logic for bookings, availability blocks, requests, shifts,
  sharing, or audit.
- Public share endpoints or share-link projection logic.
- Intake flow beyond a route placeholder.
- Notification center.
- Manager Lite surfaces.
- Payments, invoicing, marketplace, broad intake, external calendar truth.

Those belong to Phase 23+ and will build on the foundation this branch
establishes.

## Acceptance criteria

This branch is ready to merge when all 14 Phase 22 Definition-of-Done items
are met, verified against a live preview deploy by the owner. See
`docs/phases/22-canonical-beta-foundation.md` §Definition of done for the
full list.
